# End-to-End Application Audit — 2026-06-12

Full audit across all roles: buyer, customer account, admin, lab technician, inventory, commerce/money path, subscriptions, and cross-cutting security. Conducted by 8 parallel read-only review agents; every finding below was verified against actual code (file:line). Baseline at audit time: **351/351 vitest tests passing, eslint clean** on `main` @ 952f221.

Severity legend: **CRITICAL** = security hole, money loss, or compliance-chain break — launch blocker. **HIGH** = serious correctness/security defect likely to bite in production. **MEDIUM/LOW** = should fix, lower urgency.

---

## 1. Critical findings (launch blockers)

### C1. Unauthenticated privilege escalation to `founder`
`src/features/admin/team/actions/invite-user.ts:8-39` — `createInvitation(email, role, invitedByUserId)` is a `'use server'` action (publicly invocable POST endpoint) with **no auth check**; both `role` and `invitedByUserId` are caller-supplied and written via the service-role client. `acceptInvite` (`src/features/invitations/actions/accept-invite.ts:43-50`) trusts `invitation.role`. Chain: self-signup (open magic-link) → call `createInvitation(attacker@evil.com, 'founder', own_uid)` → `acceptInvite` → full founder account with access to all Rx PII, admin, and lab. Found independently by two agents.

### C2. Anyone can issue real Shopify refunds
`src/features/admin/returns/actions/review-return.ts:23-63` — no auth check at all; `reviewerUserId` is a spoofable parameter. On `approved_refund` it calls `createRefund()` against the live Shopify Admin API with a caller-supplied amount (capped only by Shopify's remaining-refundable). Found independently by three agents.

### C3. `/thanks/[orderId]` is an Rx-token oracle + email leak (IDOR)
`src/app/thanks/[orderId]/page.tsx:15-41,71` — looks up any order by guessable sequential `shopify_order_number` with zero validation, prints `customer_email`, and **mints a fresh valid 30-day Rx token** via `generateRxToken(orderId)`. Enumerating order numbers yields each customer's email plus a live link to upload/replace their prescription and start returns. The Rx token HMAC itself is sound — this page hands valid tokens to anyone.

### C4. Cart attribute key mismatch silently kills the entire Rx chain for storefront orders
`src/app/checkout/route.ts:19` sends `{ key: 'lens_type', ... }`; `src/lib/commerce/sync.ts:209` only matches `'lenstype'`/`'_lenstype'`. Every storefront Rx order therefore mirrors as `has_rx_items=false, rx_status='none'`: no day-0 Rx email, no "upload your prescription" CTA on /thanks, never enters the awaiting-Rx queue, reminders never fire — and the order can never be fulfilled (work orders require Rx approval). The shipment compliance gate still holds, but the paid order is a dead end. No test covers attribute parsing. Found independently by two agents.

### C5. Lens upgrade prices are hardcoded client-side and never charged
`src/features/shop/lens-options.ts:10-27`, `PdpConfigurator.tsx:37` — `priceDelta` values ($50 SV, $150 progressive, $30 AR, $85 photochromic, …) are summed into the displayed cart total, but checkout sends only the frame `variantId`; lens choices ride along as informational attributes that don't affect Shopify pricing. Customer sees $195, Shopify charges $145. Revenue leak + direct violation of the "Don't hardcode prices" rule. Lens upgrades currently have **no commerce representation in Shopify at all**.

### C6. `orders/updated` webhook clobbers Rx review state
`src/lib/commerce/sync.ts:276,295-300` — on update, the full row including a recomputed `rx_status` (`awaiting_upload`) is written, overwriting `uploaded_pending_review`/`approved`. Any orders/updated event (fulfillment, refund, tag edit) or each 48h reconcile-cron pass resets review state, corrupting the admin Rx queue and reminder pipeline. No DB trigger protects `orders.rx_status`.

### C7. End-of-term refund mode silently confiscates customer money on Shopify API failure
`src/app/api/cron/membership-expiry/route.ts:100-106` + `src/features/subscriptions/lib/end-of-term.ts:111-134` — `fetchCapturedAmount` catches errors and returns `0`, so `computeProRataRefund` → 0, `createRefund` is skipped, yet the membership is still marked `refunded` and slots expired, counted as success. A transient Shopify outage on the daily tick = customers permanently lose unredeemed pairs and the refund, with zero error surfaced. (`cancelMembership` aborts correctly on the same failure.)

### C8. Non-atomic inventory reserve can oversell and permanently desync the ledger
`src/features/subscriptions/actions/start-redemption.ts:157-190` — read `pool_quantity` → JS check → write `qty-1`. Two concurrent redemptions of the last unit both pass: two reservations for one unit, pool decremented once while the ledger records two `-1`s (drift the `>=0` constraint can't catch). All `+1` release paths share the same lost-update shape. There are **no SQL RPCs / atomic decrements anywhere** in the inventory path.

## 2. Systemic root cause: server actions are an unguarded trust boundary

`src/middleware.ts` only matches `/admin/:path*` and `/lab/:path*` pages and only checks **cookie presence** (acknowledged in a comment). Real auth lives in page layouts and inside individual actions. The correct pattern exists and is applied in `reviewRx`, `cancelMembership`, `savePlan`, `releaseWorkOrder`, `moveJob`, `startRedemption`, all `admin-membership-ops`, and all lab actions — but these `'use server'` actions skip it entirely:

| Action | File | Impact |
|---|---|---|
| `createInvitation` | `features/admin/team/actions/invite-user.ts:8` | **C1** — mint any-role invite |
| `reviewReturn` | `features/admin/returns/actions/review-return.ts:23` | **C2** — real refunds |
| `adjustInventory` / `pushInventoryToShopify` | `features/admin/inventory/actions/adjust-inventory.ts:9,59` | rewrite stock, push to live Shopify |
| `createDrop` / `updateDrop` | `features/admin/drops/actions/save-drop.ts:20,41` | edit public storefront content |
| `generateWorkOrder` | `features/admin/actions/generate-work-order.ts:31` | unauthenticated + **no dedup** → duplicate work orders/lab jobs (no UNIQUE on `work_orders.rx_file_id`) |
| `submitRx` | `features/rx-intake/actions/submit-rx.ts:34` | no token re-check; trusts `orderId`/`storagePath` (path not bound to order) |
| `requestReturn` | `features/returns/actions/request-return.ts:30` | anyone with an order UUID can file returns; racy RMA counter |

**Fix pattern:** every `'use server'` export calls `getCurrentUser()`/token verification first, derives `user.id` from the session, and never accepts `userId`/`reviewerUserId`/`invitedByUserId` as a parameter.

## 3. High findings

**Security / data**
- `subscription_plans` + `subscription_addon_options` have **RLS disabled** (`00029_subscription_core.sql` enables it only for memberships/redemptions) → readable *and writable* via the public anon key through PostgREST.
- `/api/cron/reconcile` **fails open** when `CRON_SECRET` unset and uses non-timing-safe `!==` (`reconcile/route.ts:13-18`); the other three crons fail closed with `timingSafeEqual`.
- `/track/[orderId]` (`page.tsx:21-26`): unauthenticated order enumeration — fulfillment + Rx status for any order number. `/api/rx/order-status` is a similar existence oracle.

**Commerce / money**
- `createFulfillment` posts to the **legacy order-scoped REST endpoint removed in Shopify API 2023-04** while pinning `2025-01` (`shopify-admin.ts:55-71`); failure is swallowed by `console.error` (`create-shipment.ts:153-174`). In production: Shopify orders stay unfulfilled and customers never get the tracking email — silently. Tests mock the call, so they can't catch it. **Needs a dev-store verification, then migration to fulfillment-orders API.**
- Line-item sync does delete + reinsert on every order webhook (`sync.ts:320-342`); `work_orders`, `rx_files`, `returns`, `subscription_redemptions` all FK those rows without cascade → once referenced, delete fails (logged only) and reinsert **duplicates line items on every subsequent webhook**.
- Cart prices are frozen into localStorage at add-to-cart and never revalidated (`CartContext.tsx:84-87`); displayed subtotal can diverge from what Shopify charges.
- `cartCreate` never inspects `userErrors` (`shopify.ts:302-329`; the mutation doesn't even select them) → stale/deleted variant = opaque 500 dead-end at checkout with no recovery path.

**Subscriptions**
- Second membership purchase by an existing member: both unique constraints surface as `23505`, treated as benign idempotency (`provision-membership.ts:83-90`) → customer pays, gets nothing, no audit row.
- Paid-but-never-advanced add-on black hole: 60-min TTL vs long-lived Shopify checkouts; late payment hits `not_pending_payment` and is dropped — money captured, no order, no admin surface (`confirm-addon-payment.ts:58-60`).
- Sweep ↔ confirm race: `sweep-abandoned.ts:30-41` resets slots without re-checking `status='pending_payment'` in the WHERE; interleaving with `confirmAddonPayment` yields a live order + re-redeemable slot + phantom stock unit.
- Non-Rx (sunglasses/plano) redemption dead-end: always set to `awaiting_rx` (`start-redemption.ts:207-214`) but the synthesized order has `rx_status='none'` → never enters the Rx queue → never ships, **and** `awaiting_rx` is a committed status, blocking the membership's end-of-term/refund/cancel forever. Spec'd skip-to-work-order (core §4.7) is unimplemented.
- Refund webhook dead-letters permanently if a committed slot is in flight >48h (guard trigger raise → 500 → Shopify gives up retrying; nothing replays `webhook_events`).

**Lab**
- `addQcPhoto` accepts any string (`add-qc-photo.ts:7-29`) — QC compliance gate satisfiable with `'fake'`; photos attachable to the wrong job; read-modify-write race loses concurrent uploads.
- Lab can fetch the work-order PDF for **any** work order including unreleased ones (`api/work-orders/[id]/pdf/route.ts:9-24`); Rx-PII reads are never audit-logged.
- Production can start before admin release: only the `qc`→ and →`ship` transitions are guarded (`move-job.ts:30-51`); `inbox` → `ready_to_cut`/`on_edger` is free.

## 4. Medium findings (condensed)

- PDF archival writes to nonexistent bucket `work-orders` (migration creates `work-order-pdfs`); error unchecked, `pdf_storage_path` set anyway — silently broken forever (`pdf/route.ts:63-71`).
- `createShipment` is non-atomic and non-idempotent: 3 unchecked writes, no uniqueness — double-click → duplicate shipments + duplicate fulfillment pushes (`create-shipment.ts:102-126`).
- Shipment compliance gate exists only in the action layer; no DB trigger/constraint on `shipments` mirrors it.
- `reviewRx` approves without verifying an image exists at that step; approve button enabled with no document; downstream gates hold but order is left `approved` with no work order and the failure only `console.error`'d (`review-rx.ts:33-85`). Also no guard against double-review → duplicate reviews/work orders.
- Rx rejection reason captured but **never emailed** to the customer; no rejection template exists.
- Admin awaiting-Rx page is read-only — **no cancel+refund control**, which CLAUDE.md rule 2a explicitly assigns to the admin.
- `orders/cancelled` mirrors as `financial_status='refunded'` + `fulfillment_status='unfulfilled'` unconditionally (`route.ts:146-155`) — lies about money/shipping state.
- No out-of-order webhook guard: delayed `orders/create` after `orders/paid` regresses financial status (`sync.ts:286-317`).
- Returns refund amount derived from mirrored pre-tax `line_total` (`review-return.ts:48`) → systematically under-refunds tax; violates the "never derive money from a mirrored price" rule the subscription paths follow.
- Currency passed lowercase (`'usd'`) to Shopify refund APIs that document uppercase ISO — unverified, would 422-block all returns refunds; needs dev-store test.
- Any partial refund (e.g. $5 goodwill) on a membership order kills the entire membership (`handle-refund.ts:45-92`).
- Reconcile cron only re-syncs orders: never provisions missed memberships, never replays dead-lettered webhook events, no inventory reconciliation at all.
- Grace-period customers are shown "Use a pair" but `startRedemption` rejects non-`active` (`start-redemption.ts:84-86`) — advertised-but-broken.
- Dispute on an expired/refunded membership is a silent no-op (no audit row) (`handle-dispute.ts:45-47`).
- `locked` slots strand reserved inventory forever if the flow dies mid-redemption: sweeper only covers `pending_payment`; releases filter `pending_payment` while expiry transitions `locked` slots too (`release-reserved-slots.ts:29-31`).
- "Push to Shopify" overwrites Shopify's (correctly decremented) level with our stale pool count; regular Shopify sales never decrement our pool — structural two-way drift with no reconciliation.
- Server-side Rx upload size/MIME not enforced (advisory only; bucket has no `file_size_limit`).
- Lab post-upload preview calls `/api/lab/qc-preview-url`, which doesn't exist → successful uploads reported as failures (`LabWorkOrderDetail.tsx:137`).
- Lens-blocker alignment guide renders a fabricated default PD of 63mm with no warning when PD is missing (`LabWorkOrderDetail.tsx:86-88`).
- Cart never cleared after successful checkout handoff → returning customer can re-buy duplicates.
- Home page hardcodes `dropSlug="drop-01"`; if the first drop is seeded under another slug, the primary CTA 404s for everyone (`(site)/page.tsx:138`).
- Newsletter endpoint is a silent no-op (returns success, stores nothing); waitlist + newsletter + order-status have no rate limiting.
- `customers/data_request` GDPR webhook neither registered nor handled.
- Audit-log coverage gaps: work-order generation/release, drops CRUD, invitation lifecycle, inventory adjustments (insert error ignored), lab transitions — none logged.

Low findings (token revocation, tesseract from unpkg CDN without SRI, float money math, financial-status mapping collapse, kanban link to /admin, fabricated `unlocks_at` branch no-op, etc.) are retained in the agent transcripts; ask Claude to enumerate if needed.

## 6. Role-by-role functional gaps

**Buyer:** lens upgrades have no Shopify representation (C5); no recovery UI on checkout failure; quiz→shop filter is substring matching that mismatches the catalog ("No matching frames"); sold-out default variant addable to cart; no all-sold-out PDP state.

**Customer account:** non-active/grace memberships (incl. ones with pairs still in fulfillment) render as "no active subscription"; no add-on receipts; abandoned add-on checkout offers no resume link.

**Admin:** no cancel+refund for stalled awaiting-Rx orders; no Rx-rejection notification to customer; no webhook dead-letter replay surface; no work-order dedup; returns `approved_replacement`/`approved_remake`/`approved_credit` set status but execute nothing.

**Lab:** no breakage/remake/hold flow (schema supports versions; no UI/action); **no shipping address visible anywhere in the lab UI or PDF** even though the lab ships; no batch operations/printing; no notification on release; bench checklist is unsaved client state; `lensometer_readings`/`assigned_to`/`physical_tray_qr` never written.

**Inventory:** no code path creates `inventory_pool` rows (UI claims "SKUs sync from Shopify" — they don't); regular orders/returns never touch the pool; no invariant check `pool == initial + SUM(adjustments)`; adjustment reason taxonomy unreachable (UI hardcodes `manual_correction`); redemption picker shows no stock signal.

**Subscriptions:** `in_review`/`rx_rejected`/`delivered`/`cancelled` redemption states never written (spec'd mirroring missing); no admin tool to unwind a stuck committed redemption; plan `markets` field spec'd but absent.

## 7. What's genuinely solid

- **Webhook envelope**: raw-body HMAC with `timingSafeEqual` before parsing, fail-closed, atomic insert-on-unique idempotency, 5xx-for-retry, dead-letter recording — all tested.
- **Rx compliance core**: defense-in-depth shipment gate re-checks approval/image/expiry/release/QC/destination at ship time (catches post-release revocations); DB-level invariants (`rx_file_id NOT NULL`, hard-delete block) survive future code paths.
- **Auth foundations**: roles in `profiles` (not client-editable metadata), default role `pending`, open-redirect-safe auth callback, takeover-resistant account claim, strong invitation tokens (32 random bytes, single-use, 7-day expiry), secrets hygiene clean (nothing tracked, no NEXT_PUBLIC leakage).
- **Money discipline where applied**: pro-rata base = Shopify captured amount (e44a425 verified at both call sites), refunds capped at remaining refundable, add-on confirmation pinned to webhook-verified line items and amounts — well tested.
- **Slot claiming** is the one correctly atomic conditional UPDATE in the system — the pattern the inventory paths should copy.
- **bee2e00 fixes hold**: `releaseReservedSlots` on all four expiry paths; guard-trigger raises surfaced, not swallowed.

## 8. Recommended fix order

1. **Auth sweep on server actions** (C1, C2, + §2 table) — one mechanical pattern, highest risk reduction. Add tests asserting 401/403 before DB touch (the lab actions show how).
2. **/thanks + /track token gating** (C3) — require an HMAC token query param (Shopify order-status page can carry it), stop minting Rx tokens for unauthenticated viewers, drop the email display.
3. **Attribute key fix + test** (C4) and **rx_status preservation on order update** (C6) — small diffs, restore the whole Rx pipeline; add a sync property-parsing test.
4. **Lens pricing** (C5) — decide the Shopify modeling (lens variants / bundle add-on line items) before any launch; until then remove the fake deltas from the UI.
5. **End-of-term refund abort-on-failure** (C7) — copy `cancelMembership`'s abort semantics.
6. **Atomic inventory ops** (C8 + sweep/release races) — move reserve/release to a SQL RPC with conditional UPDATE … RETURNING; add `redemption_id` to `inventory_adjustments`.
7. **Fulfillment API migration** (legacy endpoint) — verify on dev store, move to fulfillment-orders API, alert on push failure.
8. Then the HIGH/MEDIUM backlog above, roughly in listed order.

## 9. Remediation status (updated 2026-06-12, branch `fix/e2e-audit-2026-06-12`)

All 8 criticals and all HIGH findings are fixed, plus the bounded MEDIUM/LOW items. Test suite went 351 → 384 passing; lint + tsc clean throughout. Each fix is a separate commit.

**Fixed:**
- C1/C2 + §2 auth sweep — every privileged `'use server'` action now verifies session/role and derives ids from the session (createInvitation founder-only, reviewReturn, adjustInventory, pushInventoryToShopify, createDrop/updateDrop, generateWorkOrder + dedup); submitRx/requestReturn verify the Rx token and bind to the order.
- C3 — /thanks is static (no email, no token minting); /track + /api/rx/order-status require a token only delivered to the owner.
- C4/C6 — checkout sends `is_rx_required`; sync normalizes property names; orders/updated preserves `rx_status`.
- C5 — uncharged lens prices removed from the cart (frame price only); monetization is an open product decision (see memory `project-lens-upgrade-pricing`).
- C7 — end-of-term refund aborts on Shopify capture-fetch failure (no false "refunded").
- C8 — atomic reserve/release SQL RPCs (migration 00032, service_role-only); sweep↔confirm race guarded.
- HIGH — fulfillment-orders API migration; subscription_plans/addon RLS (00033); reconcile cron fail-closed; order line-item upsert (00034); provision 2nd-membership conflict + orphaned add-on payment flags.
- MEDIUM — QC-photo path binding; PDF release-gate + correct bucket + access audit; shipment idempotency; QC preview route; orders/cancelled status mapping; refund currency casing; dispute-on-terminal flag; grace-period redemption; Rx-rejection email; PD-missing warning; storage bucket size/MIME limits (00035).
- LOW — cart-clear on /thanks; cartCreate userErrors; customers/data_request GDPR webhook; dynamic drop CTA.

**Fixed since (feature work, branch `feature/non-rx-fulfillment`):**
- Non-Rx (plano) fulfillment path — replaced the blanket `work_orders.rx_file_id NOT NULL` with a conditional `requires_rx` CHECK (preserves the Rx-image invariant exactly), added `generateNonRxWorkOrder` + an admin non-Rx release queue, branched the ship gate on `requires_rx`, and routed non-Rx redemptions to a new committed `awaiting_fulfillment` status (un-stranding the membership lifecycle). Spec `docs/superpowers/specs/2026-06-20-non-rx-fulfillment-design.md`; plan `docs/superpowers/plans/2026-06-20-non-rx-fulfillment.md`.

**Deferred (feature-sized, not bugs — tracked in memory):**
- Lens-upgrade monetization via Shopify add-on line items. (memory `project-lens-upgrade-pricing`)
- Admin awaiting-Rx cancel+refund control (CLAUDE.md rule 2a) — needs a new admin action + UI.
- FDA/customs per-shipment import records (Rx rule 8) — schema columns exist, unused.
- Out-of-order webhook staleness guard; reconcile cron provisioning/dead-letter replay; `locked`-slot inventory leak on mid-redemption crash; newsletter persistence + rate limiting.

**Still needs a live/dev-store check before launch:** the fulfillment-orders API call and the refund currency casing (both exercised only against mocks here).
